# 우쿨렐레 코드 편성 고도화 — 설계 문서

- 작성일: 2026-06-28
- 대상 프로젝트: `/Users/selena/ukulele-chord-finder` (서버 0원 · 순수 정적 · 브라우저에서 Anthropic API 직접 호출)
- 한 줄 요약: 단순 "코드 찾기"에서 **레트로 스피커/카세트 감성의 우쿨렐레 편곡 + 음악 라이브러리 앱**으로 고도화. 핵심은 (1) 결정론적 우쿨렐레 운지 엔진, (2) 자료 없는 곡도 직접 편곡, (3) 구간별 스트로크, (4) 검색/스크랩북/프로필 UX 개편 + 레트로 리디자인.

## 1. 목표와 배경

### 해결할 문제
1. **운지 부정확** — 지금 운지는 `chords.js` 하드코딩 35개 + 나머지 AI 생성(가끔 틀림). → 음악이론으로 운지를 **결정론적 계산**해 근본 해결.
2. **자료 없는 곡** — 최유리 "생각을 멈추다", "당신은 누구시길래"처럼 웹에 우쿨렐레 악보가 없는 곡은 앱이 **직접 편곡 제작**해야 함.
3. **편곡 품질·스타일** — 실제 악보처럼 구간 나눔·코드 배치·인트로 진행이 자연스러워야 함.
4. **스트로크 부재** — 실제 악보엔 스트로크가 잘 없음 → **구간별 스트로크 표기**가 차별점.
5. **밋밋한 UX** — 검색/보관/프로필을 레트로 스피커·카세트 감성으로 재설계.

### 설계 원칙
- **서버 0원 · 순수 정적 유지.** 모든 외부 호출은 브라우저에서 직접(Anthropic API, iTunes Search API).
- **운지는 클라이언트가 책임** — AI는 "어떤 코드를 쓸지"만, 운지는 엔진이 계산.
- **정직한 출처 표시** — 전사/부분확인/직접편곡을 배지로 구분.
- **원키 유지** — 조옮김은 하지 않음(추후 옵션, YAGNI).
- **모바일 세로 우선.**

## 2. 참고 자료에서 확정한 관습

악보 예시(ukulele-tabs.com, 11ukul.com 등) 분석으로 확정:
- 코드는 **가사 위 인라인 배치**, 구간 라벨(Intro/Verse/Pre-Chorus/Chorus/Bridge/Solo/Outro) 사용.
- **인트로는 코드 진행만** 한 줄(예: `C G Am F`).
- **키는 상대조 병기** 흔함(예: "G major / E minor").
- **스트로크는 구간마다 다름**(Verse 다운 위주, Chorus DDUUDU 등) → "구간별 1패턴" 결정이 타당.
- 스트로크 표기: 화살표(↓↑×) 기본, 문자(DU) 호환.
- 코드표 철학: 한 코드에 여러 운지 → **가장 쉬운 자리** 선택.

> 참고 사이트는 이미지 기반이라 기계 파싱 불가. 우리가 따르는 것은 "형식·관습"이지 데이터가 아님. → 추가 학습 악보 불필요(엔진은 이론 기반, 스타일은 Claude 기존 지식 + 제시된 예시로 충분).

## 3. 아키텍처 — 컴포넌트

기존: `index.html` / `style.css` / `app.js` / `chords.js`. 다음을 추가·개편한다.

### 3.1 `chord-engine.js` (신규) — 결정론적 우쿨렐레 운지 엔진
순수 함수 모듈. 입력 코드명 → 최적 운지. **네트워크/AI 의존 0.**

- **파서** `parseChord(name)` → `{ root, quality, extensions[], bass }`
  - 지원: major, minor, dim, aug, sus2/sus4, 6, 7, maj7, m7, m7b5, dim7, add9, 9, 11, 13, 온코드(`G/B`).
  - 표기: `Bb`(♭) 우선(코드표 관습), `#`도 허용.
- **구성음** `chordTones(parsed)` → 피치클래스 집합(0–11). 텐션/생략음 규칙 포함(예: 7화음에서 5음 생략 허용).
- **운지 탐색** `findVoicing(parsed)`:
  - 표준 튜닝 GCEA(반음값 G=7, C=0, E=4, A=9), 0–12프렛 후보 전수.
  - 각 현 후보 음이 코드 구성음에 속하는 조합만 채택, 루트·3음·7음 포함도 평가.
  - **연주 난이도 점수**(낮을수록 좋음): 손가락 수, 포지션 높이(min fret), 손가락 벌림(span), 뮤트 현 수에 페널티 / 개방현·코드음 완전성에 보너스.
  - 최저 점수 1개 선택 → `{ frets:[G,C,E,A], fingers:[…], baseFret, barre?, difficulty:'easy|mid|hard' }`. -1=뮤트, 0=개방.
- **오버라이드 테이블**: 엔진 최적해와 다른 "관습적 표준형"만 소수 고정(예: D=2220 대신 흔히 2225 안 쓰게). 단위 테스트로 회귀 방지.
- **공개 API**: `UkeEngine.voicing(name)`, `UkeEngine.diatonicChords(key)`(편곡 보조용 키별 다이어토닉 코드).

### 3.2 `app.js` — 편곡 생성 개편
- **2단계 생성(generateChords) 출력에서 frets 제거.** AI는 코드 *이름* 진행 + 가사 배치 + 구간 + 키(+상대조) + BPM/박자 + **구간별 스트로크**만 출력. 운지는 클라이언트에서 `chord-engine.js`가 계산. → 출력 토큰↓, 속도↑, 정확↑.
- **소스 단계 재정의**:
  - `transcribed` — 우쿨렐레/기타/피아노 등 실제 코드 자료를 찾아 전사(코드·배치 실제).
  - `chords_verified` — 코드는 확인, 가사 배치는 추정.
  - `arranged` — 자료 없음 → **원키 기준 AI 직접 편곡**(최유리 곡 시나리오). 기존 `derived`를 대체.
- **자료 탐색 단계 명시**: ① 우쿨렐레/기타/피아노 등 *모든 악기* 코드 자료 검색 → ② 없으면 화성 이론으로 직접 편곡(`arranged`). "코드는 악기 무관" 원칙을 프롬프트에 명시.
- **우쿨렐레 적합 지시**: 원키 유지, 너무 높은 포지션 코드엔 엔진이 `difficulty:'hard'` 표시, 가능하면 흔한 우쿨렐레 셰이프 선호.
- **스트로크 출력**: 구간별 `strum` 필드(예: `"D DU UDU"` 또는 화살표). BPM·박자 참고해 추천.
- **속도 최적화**: 구조화 JSON(스키마)로 파싱 안정화, 품질모드별 `effort`/`max_uses` 유지, 1단계 식별은 그대로 빠르게.

#### 생성 JSON 스키마(개정)
```json
{
  "title":"", "artist":"", "album":"",
  "key":"C major", "relativeKey":"A minor", "capo":0,
  "bpm":92, "timeSignature":"4/4",
  "source":"transcribed|chords_verified|arranged", "sourceUrl":"",
  "intro":["C","G","Am","F"],
  "chordsUsed":["C","G","Am","F"],
  "sections":[
    {"label":"Verse 1","strum":"D - D U - U D U","lines":["[C]가사…","…"]}
  ]
}
```
> 운지(frets)는 생성 결과에 없음 — `chordsUsed`의 각 이름을 엔진에 넣어 클라이언트에서 운지 산출.

### 3.3 검색 인터페이스 (기능 1)
- 제목 입력 → 1단계 식별(`identifySong`)이 **가수·발매시기·앨범**까지 반환하도록 확장.
- 확인 카드에 앨범 표지(iTunes Search API: `term=가수 곡명`)·연도·앨범명 표시.
- 레트로 스피커 비주얼의 검색 화면.

### 3.4 스크랩북 대시보드 (기능 2)
- 기존 playlist/scrapbook 로직 재사용 + **앨범 표지 그리드**(카세트/레코드 타일).
- 카드 클릭 → 우쿨렐레 악보 + 스트로크 보기.
- 앨범 표지는 iTunes API로 곡 저장 시 캐싱(localStorage에 URL 보관).

### 3.5 My Profile (기능 3)
- **좋아요 → My Playlist**: 기존 즐겨찾기(`favorites`)를 "좋아요"로 리브랜딩, My Playlist에 모임.
- **아티스트 스토리(정적 큐레이션)**: 좋아요한 곡들의 아티스트를 인스타 스토리 UI(원형 썸네일+탭 넘김)로. 데이터는 iTunes API로 모은 **아티스트 최신 앨범/발매 소식**(실시간 IG 연동 아님). 서버 0원 유지.

## 5. 데이터 흐름
1. 제목 입력 → `identifySong`(web_search) → 가수/연도/앨범 + iTunes 표지 → 확인 카드.
2. 확인 → `generateChords`(web_search/fetch → 없으면 직접 편곡) → 개정 JSON.
3. 클라이언트: `chordsUsed`→`chord-engine.js`→운지, 가사 인라인 코드 렌더, 구간별 스트로크 렌더.
4. 좋아요/스크랩북 저장 → localStorage(+표지 URL). My Profile에서 아티스트 스토리.

## 6. 에러 처리
- 기존 `handleApiError` 유지(NO_KEY/TIMEOUT/NETWORK/401/429/400).
- iTunes API 실패 → 표지 없이 카세트/색 타일 폴백(앱 동작에 지장 없음).
- 엔진이 운지를 못 찾는 극단 코드(이론상 칠 수 없음) → 가장 근접한 부분 보이싱 + `difficulty:'hard'` 경고.

## 7. 테스트
- **`chord-engine.js` 단위 테스트(핵심)**: 알려진 코드의 기대 운지 검증(C=0003, Am=2000, F=2010, G=0232, D=2220, Dm=2210, Em=0432, A=2100, E, Bb, Cmaj7, Am7, G7, sus, 온코드 등). 회귀 방지. `test.mjs` 확장.
- 파서 엣지: `F#m7b5`, `G/B`, `Bb`, `add9`.
- 편곡 JSON 스키마 검증(필수 필드, source enum, 인라인 대괄호 균형).
- 수동: 자료 있는 곡(전사) vs 최유리 곡(arranged) 비교.
